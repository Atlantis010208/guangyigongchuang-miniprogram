# 原因
当前云函数（orders_remove/orders_update/requests_update/requests_remove）使用 `{ orderNo, userId: openid }` 进行匹配，而历史数据中的 `userId` 为 `users` 文档的 `_id`（前端仓库与监听均使用该字段）。因此更新条件不命中，返回 `updated: 0`，前端与云端数据未变化。

# 方案
- 将云函数的匹配条件改为仅按 `orderNo`（唯一）进行更新；保留 `id` 路径的情况下仍按 `doc(id)` 更新。
- 涉及函数：`orders_update`、`orders_remove`、`requests_update`、`requests_remove`。
- 不改动数据模型；后续新订单继续写入 `userId`，但更新/删除操作不再依赖 `userId`。

# 验证
- 删除与撤销操作再次触发，云函数返回 `updated > 0`；云数据库中 `isDelete=1` 或 `status=canceled` 生效，页面列表与监听过滤后消失。
