# 需求文档：灯光实景图库后端云函数

## 简介

为小程序「实景灯光图库」页面提供完整的后端云函数支持，替换当前硬编码的占位图数据。支持万级（10,000+）图片的存储、多标签分类、分页加载和检索能力。同时为管理后台提供图片与标签的 CRUD 管理接口。

## 现状分析

当前 gallery 页面存在以下问题：
- 图片数据硬编码在 `gallery.js` 的 `initialImages` 数组中（10 条占位数据）
- 使用 `picsum.photos` 外部占位图，非真实业务图片
- 每张图片仅支持单个 `category` 字段，无法满足多标签需求
- 搜索仅对 `title` 做本地模糊匹配，无后端支持
- 收藏功能仅存本地 Storage，未与云端同步
- 加载更多通过随机生成假数据模拟，无真实分页

## 需求

### 需求 1 - 图片数据存储与管理

**用户故事：** 作为管理员，我需要将真实的灯光实景图片上传到云端并关联多个标签，以便用户在图库中浏览和筛选。

#### 验收标准

1. 当管理员调用图片新增接口时，系统应将图片元数据（标题、标签数组、描述、云存储 fileID、缩略图 fileID、宽高比等）写入 `gallery_images` 集合。
2. 当管理员调用图片新增接口时，系统应自动生成冗余搜索字段 `keywords`（由标题 + 所有标签 + 描述关键词拼接而成）。
3. 当管理员调用图片更新接口时，系统应同步更新 `keywords` 冗余字段。
4. 当管理员调用图片删除接口时，系统应执行逻辑删除（`status` 设为 0），不物理删除数据。
5. 当管理员调用批量导入接口时，系统应支持一次最多 50 条图片记录的批量写入。

### 需求 2 - 标签管理

**用户故事：** 作为管理员，我需要维护一套标签体系（支持分组），以便对图片进行多维度分类。

#### 验收标准

1. 当管理员调用标签新增接口时，系统应将标签数据（名称、分组、排序权重）写入 `gallery_tags` 集合。
2. 当管理员调用标签列表接口时，系统应返回按分组归类、按排序权重排列的完整标签列表。
3. 当管理员调用标签更新接口时，系统应更新标签名称、分组或排序权重。
4. 当管理员调用标签删除接口时，系统应执行逻辑删除，且不影响已关联该标签的图片数据。
5. 系统应维护每个标签的 `imageCount` 计数字段，在图片新增/删除时自动更新。

### 需求 3 - 图库列表查询（小程序端）

**用户故事：** 作为小程序用户，我需要按分类浏览灯光图库并支持分页加载，以便高效地找到设计灵感。

#### 验收标准

1. 当用户请求图库列表时，系统应返回分页数据（默认每页 20 条），包含图片的缩略图临时 URL、标题、标签、宽高比。
2. 当用户选择某个标签进行筛选时，系统应返回 `tags` 数组中包含该标签的所有图片（利用数据库数组包含查询）。
3. 当用户同时选择多个标签筛选时，系统应返回同时包含所有选中标签的图片（AND 逻辑）。
4. 当用户请求下一页时，系统应基于游标（上一页最后一条记录的 `_id`）进行分页，避免 skip 在大数据量下的性能问题。
5. 系统应将云存储 `cloud://` fileID 转换为临时 HTTPS URL 后返回给前端。

### 需求 4 - 图库搜索（小程序端）

**用户故事：** 作为小程序用户，我需要通过关键词搜索灯光图片，以便快速找到特定场景或风格的图片。

#### 验收标准

1. 当用户输入搜索关键词时，系统应在 `keywords` 冗余字段中进行正则匹配查询。
2. 当搜索结果存在时，系统应返回分页的搜索结果列表，格式与普通列表一致。
3. 当搜索关键词为空时，系统应返回默认的全量分页列表。
4. 搜索应同时支持与标签筛选组合使用（关键词 + 标签 AND 逻辑）。

### 需求 5 - 图库收藏（小程序端，云端同步）

**用户故事：** 作为已登录的小程序用户，我需要收藏喜欢的灯光图片并在不同设备上同步，以便随时回顾。

#### 验收标准

1. 当已登录用户收藏一张图片时，系统应在 `gallery_favorites` 集合中创建一条收藏记录（关联用户 openid 和图片 ID）。
2. 当已登录用户取消收藏时，系统应逻辑删除对应的收藏记录。
3. 当用户查询收藏列表时，系统应返回该用户的所有收藏图片（分页），并包含图片详细信息。
4. 当用户查询某张图片是否已收藏时，系统应返回布尔值。
5. 当用户未登录时，收藏功能应降级为本地 Storage 存储（前端处理，云函数不涉及）。

### 需求 6 - 标签列表缓存（小程序端）

**用户故事：** 作为小程序用户，标签列表应快速加载，不需要每次都从服务器获取。

#### 验收标准

1. 当用户首次进入图库页面时，系统应返回完整的标签列表及版本号（`tagVersion`）。
2. 当管理员修改标签后，系统应自增 `tagVersion`。
3. 当用户再次进入图库页面时，前端可传入本地缓存的 `tagVersion`，若与服务端一致则返回 304 标识（无需更新）。

### 需求 7 - 数据库索引（性能保障）

**用户故事：** 作为系统，需要在万级数据量下保持查询响应时间在可接受范围内。

#### 验收标准

1. `gallery_images` 集合应创建复合索引 `{ status: 1, sortOrder: -1 }`（默认列表）。
2. `gallery_images` 集合应创建复合索引 `{ status: 1, tags: 1, sortOrder: -1 }`（标签筛选）。
3. `gallery_images` 集合应创建单字段索引 `{ keywords: 1 }`（搜索用）。
4. `gallery_favorites` 集合应创建复合索引 `{ userId: 1, imageId: 1 }`（收藏查询）。
5. 在 10,000 条数据量下，列表查询和标签筛选的响应时间应低于 500ms。
